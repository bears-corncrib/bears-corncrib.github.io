note 2: About using Expect in Bash script.

Here the problem I saw was that to use Windows paht format in expect, so I can automatically do some ssh script from Windows openssh servier(runs powershell or cmd console). The path I put in a bash var is $UTSHome="C:\Program Files (x86)\Qualcomm\QUTS\Support\python" , The challenge is to transfer the escape sign \. After some painful test, here are the final solutions.

  • Using Expect <<EOF, when $UTSHome="C:\\\Program Files (x86)\\\Qualcomm\\\QUTS\\\Support\\\python" Good tries

send "dir '$QUTSHome' \r\r"

send "dir \"${winPATHs[@]}\" \r\r"

Failed tries

send "dir '"$QUTSHome"' \r\r"

  • Using Expect <<EOF, when $UTSHome="C:\Program Files (x86)\Qualcomm\QUTS\Support\python"

    Failed tries,

send "dir \"$QUTSHome\"\r\r"

send "dir $QUTSHome \r\r"

send "dir '$QUTSHome' \r\r"

send "dir $QUTSHome \r\r"

send "dir '"$QUTSHome"' \r\r"

  • Using Expect -c ', all the bash var needs single quotation mark, when $UTSHome="C:\\\Program Files (x86)\\\Qualcomm\\\QUTS\\\Support\\\python"

    Good tries

send "dir '"$QUTSHome"' \r\r" This only work when there is no space in the path.

send "dir \"'"${winPATHs[@]}"'\" \r\r", This is good even with space, as quotation mark are moved out side of the var.

Fail tries

send "dir \"$QUTSHome\" \r\r"

send "dir '$QUTSHome' \r\r"

send "dir '\"$QUTSHome\"' \r\r"

  • Using Expect -c ', all the bash var needs single quotation mark, when $UTSHome="C:\Program Files (x86)\Qualcomm\QUTS\Support\python"

    Failed tries

send "dir '"$QUTSHome"' \r\r"

send "dir \"$QUTSHome\" \r\r"

send "dir '$QUTSHome' \r\r"

send "dir '\"$QUTSHome\"' \r\r"

The test shows that both ways have finally worked to transfer string with escape sign <1> to expect. The last, I wont to mention that Using Expect <<EOF, variables are in the same form as expect variables. This make expect hard to use for loop, like
for {set i 0} {$i<4} {incr i} { puts "$i" }, because $i is translated as bash variable, which make for loop fail. Using Expect -c ‘’ don't have this problem, as the bash variables must use single quotation marks to differenciate expect variables. Which way to use, it depenses on if you need expect variables.
